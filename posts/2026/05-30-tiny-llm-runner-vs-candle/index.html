上一篇我把 tiny-llm-runner 拆開來介紹完之後，心裡自然冒出一個問題：

Rust 圈不是早就有一個叫做 candle 的專案了嗎？Hugging Face 親兒子、CUDA / Metal 後端齊全，連 Llama / Mistral / Whisper / Stable Diffusion 都跑得起來——那我這個小小的 tiny-llm-runner 到底還有什麼存在意義？

這篇就是要正面回答這個問題。重點不是「誰比較好」——candle 在功能、效能、生態系上樣樣都把我的玩具碾壓——我想看的其實是另一件更有意思的事：當你給自己訂的目標不一樣時，最後寫出來的架構會差到什麼程度？

兩個專案都是 Rust 寫的、都吃 GGUF、都能跑 Llama，但你只要把它們的 Cargo.toml 和 src/ 並排攤開來看，就會發現它們其實在解的根本是兩回事。

先把比較的基準擺正

要公平比較，得先老實承認一件事：兩者的「設計願景」根本不在同一個量級。

維度 tiny-llm-runner candle
目標 教學版：每一行 forward pass 都要看得懂 通用 ML 框架：要當 PyTorch 在 Rust 的對應物
範圍 只跑 Llama 1/2/3 推論 跑各式架構（Llama、Mistral、Mixtral、Whisper、SD…）
後端 只有 CPU + rayon CPU（可選 MKL/Accelerate）、CUDA、Metal
量化格式 5 種（F32/F16/Q8_0/Q4_0/Q6_K） 完整 GGML 量化家族 + 自家 QTensor
訓練 沒有 有（autograd）
程式碼規模 < 2,000 行 數萬行（多個 crate）

把 tiny-llm-runner 拿來和 candle 比，大概就像拿一台「自己手焊的真空管收音機」去跟一台「現代多媒體電視」比——兩者都會發出聲音，但它們存在的理由完全是兩回事。所以底下我不打算去比效能或功能（比了也只會是一面倒），而是想看：為了達成各自的目標，這兩邊在哪些設計選擇上做了不一樣的取捨。

差異一：Tensor 抽象的厚度

最容易看出兩者哲學差異的地方，就是「Tensor 這個型別到底是什麼」這件事。

tiny-llm-runner：32-byte 的薄殼

我的 TensorView 全部就只有這幾個欄位：

pub struct TensorView<'a> {
    pub data: &'a [u8],   // 直接借用 mmap 的 bytes
    pub ggml_type: GgmlType,
    pub dims: [u64; 4],
    pub n_dims: usize,
}

它根本不是一個 Tensor——它只是「mmap 裡某段 byte 的解釋方式」而已。沒有 device、沒有 dtype 抽象（只有 GgmlType 這個列舉）、沒有自動微分的計算圖、沒有 broadcasting 規則、沒有 lazy execution。它只暴露兩個方法：dot_row 和 dequant_row，就這樣。

candle：完整的 PyTorch 風格 Tensor

candle_core::Tensor 的概念負擔則完全是另一個世界。它要追蹤：

  • Shape：可變維度的張量形狀，支援 broadcasting。
  • DType：F32、F16、BF16、U8、U32、I64、F64…
  • Device：Cpu、Cuda(...)、Metal(...)，每個張量都「住」在某一個裝置上。
  • Storage：實際資料的 reference-counted handle，支援共享和切片。
  • Op tracking：autograd 要用——每個 a + b 都會記下「我是從 a 和 b 加出來的」這條邊，反向傳播時才能還原計算圖。

換句話說，candle::Tensor 是一個有身分證的對象——它知道自己住哪、長什麼樣、打哪兒來。TensorView 呢？比較像一張臨時通行證，過了這個 scope 就什麼都不剩了。

這個差異會傳染到整個 codebase

當你的 Tensor 抽象很厚時，所有運算都得乖乖透過它走。candle 裡你寫 x.matmul(&w)，回傳的是一個新的 Tensor，背後跑的可能是 CPU 的 BLAS、CUDA 的 cuBLAS、或 Metal 的 MPS——你完全不用操心。

但當你的抽象很薄（像我這個）時，matvec 直接吃 &mut [f32] 和 &TensorView，沒有任何派發層，整個函式 inline 起來幾乎沒有 overhead：

pub fn matvec(out: &mut [f32], w: &TensorView<'_>, x: &[f32]) {
    out.par_iter_mut().enumerate().for_each(|(i, o)| {
        *o = w.dot_row(i, x);   // 直接 match ggml_type 派發
    });
}

代價當然有：這個 matvec 永遠只能在 CPU 上跑、只能算 f32×量化、只能做 row-parallel。要支援 GPU？沒辦法。要支援 batch-2 的 matrix×matrix？得整段重寫。

差異二：權重從 GGUF 到記憶體的路徑

這條路徑兩邊走得很不一樣，也是兩種哲學最具體的展現囉。

tiny-llm-runner：mmap → 不解壓 → row-by-row dot

我的 LlamaModel::load 只做兩件事：

  1. 把 GGUF 的 tensor 列表掃一遍，建一個 name → TensorInfo 的 HashMap。
  2. 對每個我需要的張量名（token_embd.weight、blk.0.attn_q.weight…），建一個 TensorView。

注意喔，整個過程沒有讀任何權重資料——TensorView 只記住「這個張量住在 mmap 的哪一段 bytes」而已。要等到 forward() 真的呼叫 dot_row(i, x) 時，才會去讀那一個 row 的 bytes、解碼、做內積。

整個生命週期裡，Q4_0 的權重就一直是 Q4_0 乖乖躺在硬碟上，從頭到尾沒被解壓成 f32 矩陣丟進 RAM。一個 Q4_0 的 7B 模型大約 4 GB，你要是解壓成 f32 就變 28 GB——這對筆電來說可是天差地別。

candle：QTensor → QMatMul → 編譯期分派

candle 的處理方式就精緻很多了。它有一個專門的 candle_core::quantized 模組，裡面有：

  • QTensor：量化張量的內部表示。
  • QMatMul：把量化權重和 f32 啟動值相乘的高階介面。
  • gguf_file::Content：解析 GGUF 檔。

載入流程大致是：

  1. gguf_file::Content::read(...) 把整個 metadata 讀進來。
  2. 對每個張量，用 tensor.qmatmul(...) 或 tensor.dequantize(...) 取得 QTensor。
  3. 把 QTensor 包進 QMatMul，當作一個可呼叫的層。

QMatMul 在 forward 時會根據量化型別和後端（CPU/CUDA/Metal）派發到對應的 SIMD/CUDA kernel。CPU 上是 AVX2/NEON 手工最佳化過的；CUDA 上則是專門的 dequant+gemm fused kernel——這誠意可不是我那個玩具比得上的。

兩條路的代價

對 tiny-llm-runner 來說，「不解壓」是一把雙面刃：

  • ✅ 記憶體用量極低、不需要 warmup。
  • ✅ 大檔案啟動秒開（mmap 立即返回）。
  • ❌ 沒有任何 SIMD/向量化——我的 dot_q4_0 是純標量迴圈，老實說比 candle 慢了一個數量級以上。

對 candle 來說，多走一層 QTensor 抽象的代價也很實在：

  • ✅ 同一份程式碼可以在 CPU/CUDA/Metal 上跑。
  • ✅ SIMD 最佳化由底層 kernel 統一處理。
  • ❌ 程式碼複雜度暴增——要看懂 QMatMul::forward 在哪個分支會做什麼事，得跨好幾個 trait、好幾個 cfg flag。

差異三：模型架構是寫死的還是組裝的

這個維度大概是最直觀的「規模差距」了吧。

tiny-llm-runner：把 Llama 寫死進去

我的 runner.rs 整個 forward pass 是一段大約 100 行的指令式程式碼，每一行都假設模型是 Llama：

// attention norm → qkv → RoPE → GQA attention → output proj
rmsnorm(&mut self.xb, &self.x, &layer.attn_norm, cfg.rms_eps);
matvec(&mut self.q, &layer.wq, &self.xb);
matvec(krow, &layer.wk, &self.xb);
matvec(vrow, &layer.wv, &self.xb);
apply_rope(&mut self.q, pos, head_dim, cfg.rope_dim_count, ...);
// ... 以下省略

要支援 Mistral？得去改 RoPE 的 sliding window mask。要支援 Mixtral？得加上 MoE 路由邏輯。每多支援一個架構，就得回去動 runner.rs 一次。

candle：用 nn::Module trait 組合

candle 走的是 PyTorch 風格：每一層是一個實作了 Module trait 的 struct，forward 是它的方法。Llama 的整個架構在 candle_transformers::models::llama 裡是這樣組起來的：

// 大約的形狀（為了清楚我簡化過）
pub struct Block {
    rms_1: RmsNorm,
    attn: CausalSelfAttention,
    rms_2: RmsNorm,
    mlp: Mlp,
}

impl Module for Block {
    fn forward(&self, x: &Tensor) -> Result<Tensor> {
        let h = (x + self.attn.forward(&self.rms_1.forward(x)?)?)?;
        let h = (&h + self.mlp.forward(&self.rms_2.forward(&h)?)?)?;
        Ok(h)
    }
}

每一層都是獨立的元件，換一個架構不過就是換個組合方式。Mistral 用的是同一套 RmsNorm 和 Mlp，只是把 CausalSelfAttention 換成支援 sliding window 的版本；Mixtral 則是把 Mlp 換成 SparseMoeBlock。乾淨吧。

VarBuilder：把名字和張量解耦的關鍵

candle 在「載入權重」這件事上有一個我覺得頗漂亮的抽象：VarBuilder。它是一個 hierarchical 的權重來源，用起來像這樣：

let vb = VarBuilder::from_gguf("model.gguf", &device)?;
let block_0 = Block::load(vb.pp("blk.0"), &cfg)?;
let block_1 = Block::load(vb.pp("blk.1"), &cfg)?;

vb.pp("blk.0") 會回傳一個「prefix 是 blk.0 的子 builder」，等這個子 builder 再去要 attn_q.weight 時，實際上找的就是 blk.0.attn_q.weight。這麼一來，模型的層級結構和權重的命名空間就自然對應上了——你不用在每一層的 load 函式裡手動拼字串。

而我的 model.rs 呢，則是另一個極端：

for l in 0..config.n_layer {
    layers.push(LayerWeights {
        attn_norm: load_f32_vec(&by_name, &format!("blk.{l}.attn_norm.weight"), blob)?,
        wq: view(&by_name, &format!("blk.{l}.attn_q.weight"), blob)?,
        // ...
    });
}

直接把 layer index 用 format! 拼進去，簡單粗暴。只是這招也就只能應付「我早就知道要找的張量叫什麼名字」這種情境而已，換一個架構，這整段都得改寫。

差異四：autograd —— 有沒有反向傳播

這大概是最大、也最一刀切的差異了。

candle 是一個訓練框架——它的 Tensor 在做運算時會自動建立計算圖，你呼叫 loss.backward() 就能拿到所有參數的梯度。這也是為什麼它的 Tensor 非得記住「我是從哪些 Tensor 來的」、非得有 reference counting、非得處理生命週期不可。

tiny-llm-runner 則是完全不知道反向傳播是什麼東西。所有運算都是純函式式的：吃 &[f32] 進去、寫 &mut [f32] 出來，算完就忘。沒有梯度、沒有計算圖、沒有 backward，乾乾淨淨。

這個取捨的影響其實比我想像中還深：

  • 沒有 autograd → 不需要 op 物件 → 不需要 trait object → 不需要 reference counting → 不需要 Arc<Tensor> → 不需要 thread-safe 設計。
  • 結果就是：我可以放心拿 &mut [f32] 切來切去，編譯器幫我擋住所有別名問題，程式碼長得像極了 C，卻完全沒用到 unsafe。

換句話說，光是「我只做推論」這一個決定，就讓整個 codebase 的型別系統瘦了一整個量級。

差異五：執行模型 —— eager 還是 graph？

tiny-llm-runner 是純 eager：每個 matvec 呼叫就是一個當下立刻發生的計算。沒有圖、沒有 fusion、沒有調度器，一個都沒有。

candle 大致上也是 eager 的（這正是它對應到 PyTorch 的地方），不過它的 Tensor 抽象有替未來的 graph 最佳化留了空間。實際上已經有些情境下 candle 會做 op fusion（例如 softmax + attention mask 在某些 backend 上就是同一個 kernel）。

這個差異的代價是：candle 在 hot path 上，每個運算都得過一層型別/裝置/dtype 派發。對小張量來說這個 overhead 不太能忽略，但對大張量（也就是典型的 LLM 推論）來說就相對微不足道了。至於我嘛——從定義上就不可能有這個 overhead，因為我根本沒有派發這回事 :-)

那什麼時候該用哪一個？

我對自己這個玩具的定位倒是很清楚：

這些情況，請用 candle：

  • 想要在 Rust 裡跑非 Llama 的模型（Whisper、SD、各種 vision encoder）。
  • 需要 GPU 加速（CUDA / Metal）。
  • 想要 fine-tune，或在 Rust 裡訓練。
  • 想要把同一份模型程式碼跑在多種 backend 上。

而寫 tiny-llm-runner 這種東西，價值在哪？

  • 教學：把每一個量化格式、每一個 RoPE 變體、每一個 attention 細節，都老老實實自己摸過一遍。
  • 可審計：forward pass 出 bug 的時候，你不用翻 30 個檔案、跨 5 個 trait——全部就攤在 runner.rs 那 100 行裡。
  • 超低依賴：整個 Cargo.toml 只有 anyhow、clap、half、memmap2、rayon，再加我自己寫的 llm-gguf-parser。沒有 BLAS、沒有 CUDA toolkit、沒有 LLVM，清爽。
  • 特殊嵌入場景：如果你想把推論塞進一個極小的環境（IoT、WASM、安全沙盒），tiny-llm-runner 這種「沒有外部依賴的 forward pass」算是個還算合理的起點。

我從這次比較中學到的事

寫完 tiny-llm-runner 再回頭去讀 candle 的程式碼，最大的收穫其實不是「我發現 candle 哪裡寫得不好」（事實上它寫得真的很好），而是讓我親眼看清楚「為了通用性，到底得付出多少抽象成本」。

candle 的每一層抽象——Tensor、Device、DType、Module、VarBuilder、QMatMul——都不是憑空長出來的，每一個都是在解一個「不抽象，使用者就會痛」的問題。但反過來想，如果你的使用情境根本不需要那種通用性（好比說「我只想在 CPU 上跑 Llama Q4_0、只做 inference」），那這些抽象對你來說就只剩下純粹的閱讀負擔了。

這也是為什麼我覺得 tiny-llm-runner 並不是 candle 的「失敗版」——它只是站在另一個座標點上的一個合理選擇罷了。抽象是手段，不是目的。 當你的目標就是「我要把 forward pass 整個看懂」時，零抽象的版本反而才是最對的設計呢。
